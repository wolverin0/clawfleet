## Root-cause finding that surfaced from dogfood

**Bug:** `manager.write(sessionId, text + '\r')` writes the text AND carriage return as
a single PTY chunk. Claude Code's TUI (2.1.112) absorbs the trailing `\r` as a newline
in the draft input buffer — the prompt accumulates but **is never submitted**. The only
reliable submit path is to send the `\r` as a SEPARATE PTY write after a ~100 ms flush
delay. This matched the rule that's been sitting in `~/.claude/CLAUDE.md` all along:

> **Always follow `send_prompt` with `send_key("enter")`.** Enter after typing is
> unreliable.

The v3.0 handoff/auto-handoff/queue/inject paths were all bundling the `\r` inline.
Symptoms the first dogfood pass surfaced:

- A2A handoff left the full instruction sitting in pane-a's draft buffer for 10 s.
  Manual `POST /key {key:"enter"}` submitted it and Claude then wrote the handoff file
  correctly.
- Auto-handoff step 1 readiness check timed out at 120 s — same cause, the check prompt
  never got submitted.
- Claude initially complained the prompt was "truncated" (because when a prompt is this
  long and sits in the draft for seconds while the TUI re-renders, early lines scroll
  off the visible viewport and the LLM response anchors on what's still on screen).

**Fix:** new `PtyManager.writeAndSubmit(sid, text, flushMs=120)` helper. Writes `text`,
awaits the flush, writes `\r`. Every orchestrator-owned injection path now goes through
this helper: `handoff-routes.ts`, `auto-handoff.ts` (readiness check + /handoff
dispatch + /clear + continuation), `pane-queue.ts` (both the pane_idle drain and the
force-drain HTTP endpoint), and `inject-context.ts`.

After the fix:
- Flow 1 submits the instruction cleanly in the same turn; Claude produces the handoff
  file without any manual intervention.
- Flow 2 progresses through all 7 steps: readiness → /handoff → 7467 B file → settle →
  /clear → continuation prompt.
- Flows 4 and 5 were already passing against cmd.exe (short one-liners — the draft
  buffer didn't have time to visibly stall) but the refactor keeps them using the same
  correctness path.

## Side-findings, filed for follow-up

1. **Auto-handoff readiness prompt says `at ${pct}%` even when `pct === 'unknown'`.**
   On a fresh Claude pane the TUI hasn't printed a `Ctx: X%` status line yet, so
   `readCtxPercent()` returns `'unknown'`. Reads a little weird but harmless; pure UX
   copy, not a defect.
2. **Broadcast fanout race** still requires waiting for BroadcastBar's 3 s poll before
   clicking. Documented in Phase 11 Playwright UI.6.
3. **wezbridge MCP is not registered in fresh Claude subprocesses spawned by
   theorchestra.** The user's global `~/.claude/mcp-config.json` (or whatever layer
   registers wezbridge) is not inherited cleanly when Claude boots inside a ConPTY
   child. Live A2A message delivery needs either:
   (a) theorchestra to generate/seed an `.mcp.json` in the spawned pane's cwd, or
   (b) the user to configure wezbridge MCP in their global Claude user-scope settings.
   Out of scope for this dogfood — surfaced for future Phase.
