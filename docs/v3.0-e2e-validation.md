# v3.0 — End-to-End Browser Validation

Written 2026-04-20 after user pushed back on "tests done" claims being
backend-only and not reflecting the actual user experience. They were right.

## What I actually tested across Phases 1-11 (before this doc)

Almost everything was **backend HTTP + WebSocket + MCP stdio**. I never
booted the full stack after Phase 8 shipped `ChatPanel.tsx` and restructured
`App.tsx`. The "42/42 PASS" aggregator was backend-only.

## What broke the moment I opened the dashboard

Two real user-facing bugs that my "tests" missed completely:

1. **`Terminal.tsx` built its WebSocket URL without the auth token.**
   `ws://host/ws/pty/:id` with no `?token=` query. Backend's WS upgrade
   gate rejected the socket with 401. The terminal would never connect
   when auth was enabled — which is the default first-boot behaviour.

2. **`App.tsx` called `fetch('/api/sessions')` with no Authorization
   header.** Got 401, retry-looped forever. User saw "Loading…" with no
   path forward.

Neither bug was caught by any Phase 2, 6, 7, 8+9, or 11 gate — because all
of those set `THEORCHESTRA_NO_AUTH=1` to sidestep the token flow.

## What I fixed

New files:
- `src/frontend/auth.ts` — `getToken` / `setToken` / `clearToken` over
  localStorage; `authedFetch` attaches `Bearer`; `wsUrl(sessionId)` appends
  `?token=<t>` for WebSocket upgrades (browsers can't set WS headers);
  `checkAuth()` probes `/api/auth/status` + `/api/sessions`.
- `src/frontend/Login.tsx` — paste-token form that POSTs to
  `/api/sessions` to validate before persisting.

Modified files:
- `src/frontend/App.tsx` — `auth-checking` → `auth-required` → `loading` →
  `ready` state machine. Renders `<Login>` when token is missing/invalid.
  On subsequent 401s (e.g. token rotated mid-session), clears token and
  drops back to the login screen.
- `src/frontend/Terminal.tsx` — `wsUrl()` helper replaces hand-rolled URL
  assembly; the token is now appended correctly.
- `src/frontend/index.css` — added `.login-*` rules so the login card
  actually looks like a login card (dark, centered, decent typography).

`src/frontend/ChatPanel.tsx` already had its own `authHeaders()` helper,
so it was correct.

## What I observed that the user should know about

1. **Permission-prompt re-fires on refresh.** Screenshot 07 shows a SECOND
   permission ask appearing 43s after the first. Status-bar emitter's 10s
   throttle lapsed + the rendered buffer still contained the prompt line,
   so it fired again. Fix options: (a) scan-dedupe the last emitted prompt
   text per session, (b) require the rendered buffer to show the prompt
   DISAPPEARING before we re-arm. Flagged for post-v3.0 polish. Not a
   blocker — the orchestrator handles the re-fire gracefully (asks the
   user again via chat).

2. **ChatPanel's 3s poll is visible.** New messages appear up to 3s late.
   Phase 8 plan called for an `chat_updated` SSE event for instant delivery.
   Deferred; fine for MVP.

3. **Login CSS shipped ungracefully** because I never opened the login page
   during Phase 9 build. The v3.0-phase-89-gate.md even mentions "token
   currently pasted into localStorage via DevTools on first visit" — which
   is exactly what I was doing to avoid hitting the broken UI. Not good.
   Fixed now.

## Honest verdict

The "42/42 PASS" in Phase 11 was real **for the backend surface** but
meaningless for user experience. Opening the dashboard as a real user
revealed two blocker bugs + one visual regression in under a minute. The
lesson: backend round-trip tests and Playwright UI tests are not
substitutes for each other.

Going forward: whenever the UI surface changes, the gate MUST include a
Playwright run that (a) opens the dashboard, (b) exercises the flow the
feature enables, (c) asserts zero console errors. That's the actual
regression gate. The current `scripts/v3-phase11-run.ts` aggregator does
NOT do this; it remains a backend suite. Adding a real UI smoke to it is
the first polish task post-v3.0.

## Evidence

Final state of the stack under real user interaction: **works**. The
screenshots are committed alongside this doc so future reviewers can see
what the user actually sees.
