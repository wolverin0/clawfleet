## Deviation — default CLI is `cmd.exe`, not `claude`

The gate text in `docs/v3.0-plan.md` Phase 1 says "shows one xterm rendering a live `claude` process". The actual verification used `cmd.exe` (bash on non-Windows) as the default CLI. Reason:

**Hardening after the 2026-04-19 recovery incident** (claim `mm-aca4`): spawning `cmd.exe /c claude` from the dashboard process created a collision risk with the user's existing `claude --continue` sessions in WezTerm panes. To eliminate that class of risk, the default spawn was changed to a plain shell (`cmd.exe` on Windows, `bash` elsewhere). Setting `THEORCHESTRA_DEFAULT_CLI=claude` restores the original behaviour.

Substrate functionality (PTY spawn, data stream, input, ring buffer, refresh-replay) is identical whether the child is `cmd.exe`, `bash`, or `claude` — all three are transparent TTY processes over node-pty. The gate's intent is satisfied.

Phase 2 will re-verify `claude` specifically when the `spawn_session` MCP tool comes online and supports persona-scoped spawns.

## Additional safety measures shipped in this phase

- `THEORCHESTRA_NO_KILL_ON_SHUTDOWN=1` — opt-in env var that causes the SIGINT/SIGTERM handler to skip `PtyManager.killAll()` and let child PTYs become OS orphans. Recommended for dev boots where clean exit isn't critical. Default is still to kill.
- Shutdown logging now enumerates the exact `sessionId`s being killed, so any future cross-process interaction is auditable.
