## Context

Phase 3 builds the active-orchestrator input pipeline: push-based SSE events + server-side deterministic terminal rendering. The original plan specified a Rust `agent-browser` CLI spawning headless Chromium to read xterm.js's a11y tree; an ADR-003 addendum pivoted to `@xterm/headless` (same VT parser as the frontend, pure Node, no binary distribution, no cross-process hop).

## Deviations from the plan

1. **No Rust agent-browser CLI.** ADR-003 addendum replaces it with `@xterm/headless`. Benefits: no binary distribution, no cross-process hop, sub-millisecond reads, deterministic against what the browser renders.

2. **Stuck emitter uses render-based working detection, not `PtyManager.status()`.** The status heuristic flips to 'idle' after `WORKING_THRESHOLD_MS = 3000ms` of silence — which is exactly the condition under which pane_stuck should fire. Added `isVisuallyWorking(tailLines)` inside the stuck emitter with the same spinner+verb regex the status-bar emitter uses (duplicated here intentionally to avoid cross-emitter coupling).

3. **Stuck emitter poll interval is configurable.** `attachStuckEmitter(manager, bus, { thresholdMs, scanIntervalMs })` for gate-harness determinism. Default still the plan's 30s.

4. **Test-only injector `_injectForTest(id, chunk)`.** Underscore-prefixed to flag it as test-only, bypasses the child PTY to deterministically drive rendered output. Used exclusively by gate harnesses.

5. **Early-fire hazard in gate runs documented.** Because the stuck emitter keeps scanning while earlier expectations run, a spinner injected for the pane_idle test could trigger pane_stuck before that expectation subscribes. The rewrite of the pane_idle trigger (7 non-empty filler lines + ❯) keeps the spinner inside last-5 during the idle test, then pushes it out cleanly — which also re-arms the stuck emitter for the later pane_stuck test. If we ever add more "working" events to the middle of the gate sequence, verify the same re-arm rule still applies.

## TypeScript improvements forced by Phase 3

- Distributive Omit helper in `src/backend/events.ts`:
  ```ts
  type DistributiveOmit<T, K extends keyof T> = T extends unknown ? Omit<T, K> : never;
  ```
  Without this, `Omit<SseEvent, 'id'|'ts'>` collapses the discriminated union to its common fields, dropping variant-specific ones like `taskId`, `crossed`, `deadPeer`. Every emitter now publishes type-narrowed payloads.
- The B1 + B2 agents independently introduced a local `PayloadOf<T> = Extract<SseEvent, {type: T}>` helper inside their files to satisfy the SDK's strict `publish` inference. Both compile clean; not worth consolidating.

## Follow-ups surfaced during Phase 3

1. `omni-watcher.cjs` stays in-tree during dogfood; Phase 10 removes it.
2. The `peer_orphaned` event uses A2A envelope session_ids directly — when v2.7 panes publish envelopes with numeric `pane-N` IDs, the v3.0 a2a-scanner won't map them to v3.0 `SessionId` UUIDs. Acceptable for pure-v3.0 setups; Phase 10 dogfood will tell us if we need a bridging alias table.
3. `pane_stuck` fires once per working-streak — when the UI needs to show "still stuck for N minutes", it should ask `/api/sessions/:id/status` directly rather than wait for a repeat event.
4. AttachConsole warning at tsx boot (node-pty internal on Windows) is cosmetic; filed in claim `mm-aca4` context but does not block any functionality.
