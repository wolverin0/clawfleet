## What shipped

- `src/backend/session-manifest.ts` — SessionManifestStore (JSON per session in `vault/_sessions/`), attachManifestWriter (writes on spawn/data-throttled/exit), respawnDeadSessions (scans dir, `isPidAlive(pid)` check, spawns replacement using per-CLI resume strategies, deletes dead manifest + writes new one + injects banner).
- Resume registry: `claude --continue` (handles both `claude` and `cmd.exe /c claude` shim), Codex fresh respawn placeholder, generic fresh-args fallback.
- `PtyManager.injectBanner(id, text)` — operational sibling of `_injectForTest`; writes a banner into ring buffer + headless + 'data' event without touching `lastDataAt`, so emitter streak state is unaffected.
- `PtySpawnEvent` now carries `spawnArgs` so the manifest writer can persist the exact argv used at spawn.
- `scripts/theorchestra-start.ts` wires the manifest store at boot and calls `respawnDeadSessions` BEFORE the default session spawns — so cold-start recovery happens first, then the default pane is added fresh.
- Multi-device WS broadcast was already functional by construction (per-socket `manager.on('data')` listener pattern from Phase 1); Phase 6 gate proves it with a 2-client test.

## Deviations

1. **Codex resume strategy is a placeholder.** The plan says Phase 6 records the exact resume command per-CLI. Codex's `--resume <name>` flag requires a session-name field we don't yet capture at spawn time — Phase 7 or Phase 10 dogfood will wire that; for now Codex panes get fresh respawn like generic shells (acceptable fallback per the plan's risk register).
2. **No xterm SerializeAddon client-side state replay.** The spec allows it; Phase 1's ring-buffer replay already handles refresh. Adding SerializeAddon is a client-side polish nicety that doesn't affect the durability guarantee — deferring to Phase 8 frontend chat work or later.
3. **Manifest throttling is 5s on `data` events.** High-throughput panes would rewrite the manifest every 5s (just updating `lastOutputAt`). Acceptable. If this becomes a disk-I/O concern we can batch via a debounced timer.
4. **Respawn banner uses `injectBanner`**, not real PTY data. The banner is visible in rendered scrollback via the headless terminal but doesn't pass through the child shell — intentional, so the shell's own output isn't polluted.

## Follow-ups

1. Codex session-name capture for real `--resume` (Phase 7 or 10).
2. Multi-device input serialisation is natural via single-threaded event loop — if we ever need stricter ordering (e.g. partial-line writes with newline commits), add a write queue per session.
3. Consider recording the PRD role-name on the SessionRecord so post-reboot UIs can re-group respawned sessions by team.
