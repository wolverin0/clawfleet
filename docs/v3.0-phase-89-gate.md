## What shipped

### Phase 8

- `src/frontend/ChatPanel.tsx` — polls `GET /api/chat/messages`, renders orchestrator asks + user replies, handles unresolved orchestrator asks with an inline reply field, and has a bottom "user-initiated message" input. Respects `localStorage.theorchestra.token` for the bearer header.
- `src/frontend/App.tsx` restructured to a flex split: Terminal left, ChatPanel right (400px). Responsive stack on narrow viewports with a toggle.
- `src/frontend/index.css` updated with chat-bubble styles matching the existing dark theme.
- `src/backend/telegram-push.ts` — `TelegramPusher` class + `configFromEnv()`. Fire-and-forget `notify(title, body)` hits the Telegram Bot API sendMessage endpoint; no SDK, no polling, no bot listener. No-op when `TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID` aren't set.
- `src/backend/orchestrator/chat.ts` accepts the optional `TelegramPusher` and fires `notify` on every orchestrator `ask()`.
- `scripts/theorchestra-start.ts` loads Telegram config at boot and threads it into the orchestrator.

### Phase 9

- `src/backend/auth.ts` — `AuthStore` (file-backed 256-bit CSPRNG token at `vault/_auth/token.json`, chmod 600 on POSIX), `extractBearerFromHeader`, `extractTokenFromUrl`, `isExemptFromAuth`. `verify` is constant-time via `crypto.timingSafeEqual`.
- `ws-server.ts` — middleware-like gate inserted at the top of `makeHttpHandler`; applies to every non-exempt path and to WS upgrade (accepts either `Authorization` header or `?token=<t>` on upgrade URL). New endpoints: `GET /api/auth/status`, `POST /api/auth/rotate`.
- `scripts/theorchestra-start.ts` generates a token on first boot, prints the first 6 chars, exports `THEORCHESTRA_TOKEN` into the child process env so spawned MCP tools inherit it automatically. `THEORCHESTRA_NO_AUTH=1` opt-out for dev/gate runs.
- `bin/theorchestra.js` — CLI with `start [--port n]`, `stop`, `rotate-token`, `version`, `help`. `rotate-token` writes a new 256-bit token in place, preserving `createdAt` and stamping `rotatedAt`.
- `package.json` — `"bin": { "theorchestra": "./bin/theorchestra.js" }` so `npm install -g` installs the CLI.

## Deviations / follow-ups

1. **No `/login` page yet** — the ChatPanel reads the token from localStorage silently. First-install UX currently relies on the user reading the token from the server's boot log and pasting it into localStorage via DevTools. A proper `/login` route lands in Phase 10 or as a Phase 11 polish item.
2. **Telegram push is outgoing-only.** Incoming replies (user typing into Telegram) still route through v2.7's `telegram-streamer.cjs` during the dogfood period. Bidirectional v3.0 native needs a bot poller — Phase 10 scope.
3. **CLI `stop` is advisory.** It prints platform-specific `kill` instructions rather than tracking a PID file. Good enough for MVP.
4. **No auth persistence on rotate CLI path.** The inline `rotate-token` CLI rewrites the token file directly rather than calling the running backend's `POST /api/auth/rotate`. Consequence: if the backend is running and you rotate via CLI, the running process keeps using the OLD in-memory token until restart. Phase 10 dogfood: fix to call the HTTP endpoint when a backend is detected.
